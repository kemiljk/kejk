Project Nimbus

What is it?

Project Nimbus website started after we noticed the significant amount of time it took to launch a website for one of our clients. This issue occurred at both the initial launch phase as well as during any deployment process from Dev > Staging > Live. This website aims to democratise the process of website capture and deployment so that Project Managers are able to do staged deploys without requiring dev support.

My role

I was involved in every aspect of the Project Nimbus website, from conception of the overall idea, to performing interal stakeholder whiteboard sessions, creating wireframes, prototypes and working with the lead developer and project manager to ensure that the functional requirements were inline with the needs from both a development perspective as well as a project management perspective.

My role consisted of the following:

  • Research based on stakeholder whiteboard session
  • Identification and consolidation of core requirements spec
  • Creation of simple user journey diagrams to identify the pain points of developers and project managers
  • Development of information architecture
  • Creation of low-fidelity wireframes
  • Creation of high-fidelity wireframes and prototype
  • Creation of full designs, including a design system and component set
  • Provision of files and functional spec to development

Summary (the TL;DR version)

Out internal project teams regularly have to screenshot entire websites, including all popups and modal information to create a consolidated PDF that captures the state of the site at the point of approval. This is a legal and compliance issue for our clients and is an absolute requirement of all digital jobs due to the fluidity of development.

We focused on evaluating the core problems and created a wishlist that covered versions 1-1.4 based on prioritisation. The intial featureset will include the ability to trigger a screenshot capture of an entire website with all popups triggered and untriggered. It will also show visibly what IP address the site is currently active on so we know when the site hits the live DNS. The project team will also be capable of pushing the sites between Dev, Staging and Live via simple boolean toggles, check off a common pre-launch checklist, see the site's Google Lighthouse usability score, and both run a browser test on common browsers to get a pass/fail response and also create a sitemap via a single click.

There's a lot here, but these are all small–medium tasks that we regularly repeat, so creating automated tools to perform these will save countless hours. We estimated that a project manager spends around 6 hours a day completing each of these steps when required. This saving outweighs the initial project build time by many times and was therefore given the greenlight by the management team.

This project is a work in progress.

Who was it for?

The internal project management team

What challenges did they have?

Fundamentally, the challenges were a lack of time. Our project team are worked to the bone daily and can't allocate the time and resource necessary to complete all of these task manually. The secondary problem is that our internal digital team consists of my myself only and I can't allocate this much spare time for every project. All external development teams are external from the organisation and as such, the cost of performing these tasks regularly is high.

What might they need?

Our team need a large amount of things to get their job done for every major and minor update to a web project.

We discovered the following in our assessment:

  • Capturing screenshots
  • Know what tasks need to be completed at each stage
  • Deploy the sites to Dev, Staging and Live
  • Show current live IP address
  • View a usability score
  • Run a browser test
  • Create a sitemap
